### High Priority — V2 login broken with `need_captcha` (1)
- [x] **Root cause (2026-06-15):** TickTick's `POST /api/v2/user/signon` returns HTTP 500 `errorCode: need_captcha` — an anti-bot **captcha wall on the password login**. V1 (OAuth) still returns 200, so the *account* isn't banned; only the password-login path is blocked. It's worsened by a **crash→retry storm**: when V2 auth fails, `initialize()` raises → `lifespan` crashes → every incoming MCP request re-runs lifespan and re-attempts signon (logs show bursts every few seconds), which keeps the captcha flag hot. NOT 2FA (`requires_2fa: False`), NOT 6-month expiry (server is ~2 months old). (2026-06-15)
- [x] **Fix 1 (safe, helps everyone): stop the signon storm.** `UnifiedTickTickAPI.initialize()` no longer crashes on V2 failure — records a 6h cooldown timestamp + reason, logs explicit guidance (incl. "redeploy bypasses cooldown"), and starts the server in V1-only degraded mode. V2-only tools return `TickTickAPIUnavailableError` (the existing per-method guards do their job once init stops crashing). (2026-06-15)
- [x] **Fix 2 (A): set a stable `TICKTICK_DEVICE_ID`.** Loud startup warning when ephemeral; settings exposes `device_id_is_ephemeral`. README updated. (2026-06-15)
- [x] **Fix 3 (token fallback): inject a pre-obtained V2 session token + cookies** via `TICKTICK_V2_TOKEN` + `TICKTICK_V2_COOKIES`. Used as a *fallback* only — password sign-on is still tried first, so the operator can verify whether the token bypass is actually necessary by reading the logs. Browser-grab instructions added to README. (2026-06-15)
- [x] **Graceful degradation:** run V1-only when V2 is captcha-blocked instead of crashing the whole server. (2026-06-15)
- [x] **V1 OAuth expiry hint:** runtime 401s on V1 now log explicit "run `ticktick-sdk auth` and refresh TICKTICK_ACCESS_TOKEN" guidance instead of a generic auth failure. (2026-06-15)
- [x] **`ticktick_auth_status` diagnostic tool (2026-06-15):** live V1/V2 health check (lightweight read pings, catches mid-session expiry) with a plain-English verdict + the exact env var to fix. Exposes no secrets (booleans + masked device id only). Also: actionable auth-failure messages on every V2/V1 401, device-id format validation at startup, and `_v2_auth_method` tracking (password vs cookie).
- [x] **Recurrence diagnosis (2026-06-26): the failure mutated from `need_captcha` (HTTP 500) to `429 Too Many Requests`.** Live logs show the server (re)starting repeatedly (09:06:17 and 09:13:15 UTC, each aligned with a burst of MCP activity), and **every** restart re-runs `POST /user/signon` (no session persistence yet). TickTick's anti-bot now answers the repeated logins with `429`, and that same rate-limit **also blocks the cookie fallback's `GET /user/status` verification**, so the escape hatch we built (Fix 3) is itself throttled. Net: V2 is hard-degraded, and the whole daily-brief routine (all V2-routed) is dead. Likely restart cause: Railway app-sleeping/scale-to-zero waking on each activity burst, a crash loop, or repeated redeploys (⚠️ user to confirm in Railway). The cooldown timestamp is our own informational `startup + 6h`, not TickTick's real window. Cookie is probably NOT stale (user fetched it recently and is still logged in on laptop); the `429` was masking its true state.
- [x] **Made the error logs/messages accurate about 429 vs 401 (2026-06-26).** Added `_is_rate_limit_error()`; the cookie-fallback failure, the recorded `_v2_unavailable_reason`, the per-call degraded-mode error (via new `client.degraded_reason`), and the `ticktick_auth_status` verdict now distinguish a rate-limit (HTTP 429, "stop restarting and wait, do NOT refresh the cookie") from a genuinely stale session (401, "refresh `TICKTICK_V2_COOKIES`"). Previously every fallback failure was reported as "probably stale, refresh it", which sent us chasing the wrong fix. Messaging only, no auth-flow behavior change. Existing degraded-mode tests still green (verified by hand; pytest deps not installable in this container). (2026-06-26)
- [ ] **Fix the signon storm at the source (behavioral, proposed, needs approval).** Two complementary changes: (1) **Try the cookie fallback FIRST when `TICKTICK_V2_COOKIES` is set**, before password sign-on, so cold starts stop hammering `/user/signon` (the cookie path makes no login call, so it can't trip the anti-bot). Reverses the deliberate password-first order from Fix 3, whose diagnostic purpose is now moot. (2) Optionally **don't hard-fail the cookie on a 429** (treat it as transient / keep retrying per-call) instead of clearing the session and going fully degraded. Pairs with the persistent-session-cache item below.
- [ ] **Investigate why the server restarts so often (infra, user-side).** If Railway has app-sleeping/scale-to-zero or replicas >1, each wake/replica re-signs-on. Disable sleeping or add a keep-alive, or scale to a single always-on instance. This is the highest-leverage non-code fix and may let the throttle clear on its own.
- [ ] **2FA handling (for other users):** detect TickTick two-step verification at signon and surface a clear message / support a TOTP or token path. (This user has no 2FA, but others might. Current code already raises a `TickTickSessionError(requires_2fa=True)` which is picked up by the graceful path — but the operator-facing message could be more specific.) **Design note:** the **token path is effectively already done** via `TICKTICK_V2_COOKIES` (log in through a browser, completing 2FA there, then reuse the session — this skips `/user/signon`, so it sidesteps 2FA entirely). A **TOTP path** would need a `TICKTICK_TOTP_SECRET` env var so the headless server can generate the 6-digit code itself; `SessionHandler.authenticate_2fa` is the implemented-but-not-yet-wired scaffolding for that final verify step.
- [ ] **Persistent session cache (C):** optionally persist the V2 session across restarts (e.g. Railway Volume) so signon happens ~twice a year instead of on every redeploy. Document the cross-user setup implications.
- [x] **Lifespan cleanup:** initialize `client = None` before the try, drop the `if "client" in locals()` test. (2026-06-15)

- [ ] **Phase 2 — make the core daily-brief task ops survive a V2 outage.** Traced the productivity-prompt workflow end to end: every read (`list_all_tasks`→V2 `sync()`, `list_completed_tasks`, `list_abandoned_tasks`, `search_tasks`) and every write (`batch_create_tasks`, `batch_update_tasks`, `set_task_parents`) is V2-only or hard-requires V2 — so a captcha wall kills the *entire* brief, not just habits/focus. Keep V2 as the default; fall back only when V2 is down. Tradeoff to accept: V1 data is thinner (no tags / subtask links / pinned flag). Stays V2-only regardless: abandoned-task list, subtasks, tags, folders, columns. Build on Phase 1.
